## 一、本次升级带来的改动

> **状态更新（2026-05-05）**：A1-A5 与 B1 骨架已在当前代码落地；普通
> `recalculate_downstream` 入口默认**不重复执行调用方传入的 seed 列**，只有
> `patch_constant` 这类“常量变更触发精准 DAG”场景才会显式执行 seed 公式。

### A 系（默认启用，零业务接口变化）

1. **A1 索引**：`ensure_project_migrations()` 自动补建：
   - `_dependency_graph(to_table,to_column)` / `(from_table,from_column)`
   - `_formula_registry(table_name,column_name)`
   - `_cell_provenance(table_name,column_name)`
   - `_calculators(table_name)`、`_constants(name_en)`
   - `_perf_log(op,created_at)`
   - 业务表/lookup 表：`ensure_calculator_indexes(conn)` 根据 `_calculators.axes_json`
     自动建复合索引；`level/grain` 语义轴会优先使用其**实际 source 列**
     （如 `level -> lvl`）参与建索引；`register_calculator()` 也会触发一次同样补建。
2. **A2 最小列加载**：`load_table_df(conn, table, columns=...)` 接受白名单；
   公式执行前按 `parse_formula_refs / parse_row_refs` 自动裁剪。
3. **A3 批量回写**：所有逐行 `UPDATE` + provenance upsert 改为 `executemany`
   批量提交；DAG 重算内会把多个节点的写入合并到**单次 SQLite commit**。
4. **A4 calculator 批量 lookup + 缓存**：公式执行上下文内为 `_calculators`
   元数据与 `(name, kwargs)` 调用结果各自维护一层缓存，并把已解析的 metadata
   直接传入 `call_calculator()`，避免整列重算时重复读取 `_calculators`。此外，
   `VLOOKUP/XLOOKUP/MATCH` 的**精确匹配路径**也会对同一批次内的 lookup 数组建立
   预索引，避免 1000 行 × 每行线性扫数组的退化。
5. **A5 DAG 重算**：新增 `recalculate_downstream_dag(conn, seeds)`，BFS
   收集受影响节点 → Kahn 拓扑 → 按层执行 → 每节点仅一次。
   若某前置节点失败，其下游会标记为 blocked 而不是基于旧值继续写回；
   同表 `row` / `row_template` 公式也已写入 `_dependency_graph`，可参与 DAG 重算。
   旧 `recalculate_downstream` 在开关开启时自动转调，但不会重复执行调用方已改动的
   seed 列；`patch_constant` 的精准重算会显式执行 seed 公式本身。节点执行后若
   **结果值未变化**，其下游会以 `upstream_unchanged` 直接跳过，避免继续传播无效重算。
   同一轮 DAG 内现在统一使用**按表+列集的窄投影缓存**复用 DataFrame；若后续节点只多要 1~2 列，缓存会直接复用已有结果并只补缺失列。进入每一层前，执行器会先把该层所有可执行节点需要的表投影按 union 一次性预热成**共享 frames**，节点执行时直接复用这批 DataFrame 引用。DuckDB 会复用单个会话，但 SQLite Scanner 仅在实验开关开启时才尝试；多个节点的业务写入、provenance 与 perf log 会一起在 DAG 末尾提交。
6. **A3/A5 补充**：列公式与 row 公式现在会先对比新旧值，只对真正变化的单元格写回业务表；
   `_cell_provenance` 仍记录本次公式覆盖到的行，保证诊断语义不丢。

### B 系（默认关闭，骨架 + 部署预备）

- 新增 `app.services.duckdb_compute`：
  - `compute_column_via_duckdb()`：默认仍走稳定的 `duckdb.connect(":memory:") +
    con.register("df", df)` 路径；若显式开启 `use_duckdb_sqlite_scanner`，才会尝试
    `LOAD sqlite + ATTACH project.db` 的实验直读路径。两条路径最终都返回
    `[(value, row_id), ...]` 走 A3 批量回写。
  - 公式翻译白名单（当前）：支持同表 / 跨表 `@<table>[col]`、`@@table[col]`、`IF`、
    `INDEX/MATCH`、精确匹配 `VLOOKUP/XLOOKUP`、`CONCAT`、四则运算、
    `min/max/abs/round/sqrt` 与字面量；其余抛 `NotSupported` 由调用方 fallback
    到 Pandas。
- `execute_formula_on_column()` 入口已加分支：开关开启 + `is_enabled()` + 翻译成功
  → 走 DuckDB；任何异常静默 fallback。
- `recalculate_downstream_dag()` 会在整轮执行内复用一个 DuckDB 会话，避免每个 DuckDB
  节点重复建连和重复加载 SQLite 元数据。
- `load_table_df()` 现在按 **table + 精确列集** 缓存；DuckDB/Pandas 节点可复用相同窄投影，
  也会优先命中已缓存的列超集，或在已有缓存基础上只补缺失列，避免旧版全表缓存的
  `SELECT *` 回归，也进一步减少跨节点重复读 SQLite。
- DAG 执行层会把**同层可执行节点**所需的投影按表合并后一次预热；当同层多个节点共同依赖
  同一张引用表但列集不同（如一个要 `v`，一个要 `w`）时，SQLite 只需要读一次 union 投影。
- DuckDB 的 `@@table[col]` 整列引用现在也会优先从这批层级共享 frames 构造数组，避免每个公式
  再单独 `SELECT rowid,row_id,col` 一次。

`backend/requirements.txt` 已将 DuckDB 改为**正式依赖**：

```

```

- 新部署或升级旧环境时，需要重新安装依赖：

  ```bash

  source venv/bin/activate
  pip install -r requirements.txt
  ```
- `perf.use_duckdb_compute` 仍默认关闭；即使依赖已安装，未开启开关时仍走 Pandas 路径。
- `perf.use_duckdb_sqlite_scanner` 默认关闭，仅建议在专项压测时单独打开验证。

最低运行环境要求：
- Python ≥ 3.9（项目已用）
- SQLite ≥ 3.24（`ON CONFLICT ... DO UPDATE` 与 `executemany` 配合，已具备）

## 四、性能开关一览

所有开关存放在每个项目库的 `project_settings` 表，键 `perf`，值为 JSON 对象：

| 键 | 默认值 | 含义 |
|----|--------|------|
| `use_min_column_load` | `true` | A2：仅 SELECT 公式依赖列 |
| `use_batch_writeback` | `true` | A3：`executemany` 批量回写 + provenance |
| `use_batch_lookup` | `true` | A4：calculator 元数据/结果缓存 + 精确 lookup 预索引 |
| `use_dag_recalc` | `true` | A5：DAG 拓扑批量重算 |
| `use_duckdb_compute` | **`false`** | B1/B2：DuckDB 向量化计算（依赖已安装，但默认仍关闭） |
| `use_duckdb_sqlite_scanner` | `false` | B6：DuckDB 直读 SQLite 实验开关，默认关闭 |

切换示例（任意项目）：

```sql
-- 关闭 DAG 重算（紧急回退）
INSERT INTO project_settings(key, value_json, updated_at)
VALUES ('perf', '{"use_dag_recalc": false}', strftime('%Y-%m-%dT%H:%M:%SZ','now'))
ON CONFLICT(key) DO UPDATE SET
  value_json = excluded.value_json, updated_at = excluded.updated_at;

-- 开启 DuckDB 计算（依赖已随 requirements 安装，默认仍需手动开开关）
INSERT INTO project_settings(key, value_json, updated_at)
VALUES ('perf', '{"use_duckdb_compute": true}', strftime('%Y-%m-%dT%H:%M:%SZ','now'))
ON CONFLICT(key) DO UPDATE SET
  value_json = excluded.value_json, updated_at = excluded.updated_at;

-- （实验）同时开启 DuckDB SQLite Scanner
INSERT INTO project_settings(key, value_json, updated_at)
VALUES ('perf', '{"use_duckdb_compute": true, "use_duckdb_sqlite_scanner": true}', strftime('%Y-%m-%dT%H:%M:%SZ','now'))
ON CONFLICT(key) DO UPDATE SET
  value_json = excluded.value_json, updated_at = excluded.updated_at;
```

读取生效状态：

```python
from app.services.perf_flags import perf_status
perf_status(conn)
# {"flags": {...生效值...}, "defaults": {...}, "overrides": {...}}
```

## 五、回退方案

| 场景 | 操作 |
|------|------|
| A4 缓存疑似产生异常结果 | `perf` 写入 `{"use_batch_lookup": false}`，立刻回到旧逐次 SQL 路径 |
| A5 DAG 调度异常 | `{"use_dag_recalc": false}`：旧"一跳直接下游"行为 |
| A3 批量写出现 SQLite 锁/唯一约束冲突 | `{"use_batch_writeback": false}`：回到逐行 UPDATE |
| A2 最小列裁剪导致引擎报"未知列" | `{"use_min_column_load": false}`：回到 `SELECT *` |
| B 任意问题 | `{"use_duckdb_compute": false}` 即时关闭；若只是 scanner 回归，单关 `use_duckdb_sqlite_scanner` 即可 |

A1 索引可永久保留 —— 只增不删；写入性能影响经压测可忽略。

## 六、DuckDB 灰度建议

1. **第 1 步**：在测试项目开启 `use_duckdb_compute = true`。
2. **第 2 步**：选 1～2 条最简单的同表四则运算公式触发重算，对照 `_perf_log`
   中 `engine=duckdb` 与 Pandas 路径的耗时与结果。
3. **第 3 步**：把存量公式逐步迁移到 DuckDB 白名单形态（避免 `call_calculator`
   出现在重算热点公式里；可改为先用 calculator 算成中间列，再做四则运算）。
4. **第 4 步**：在生产小项目验证一周后再扩到大项目。
5. **任何时候**：关闭开关即彻底退出 DuckDB 路径，业务表与公式注册不受影响。

### 当前最值得迁移到 DuckDB 的公式形态

1. `INDEX(@@src[value], MATCH(@t[key], @@src[key], 0))`
2. `VLOOKUP(CONCAT(@t[level], '_', @t[type]), @@src[row_id], @@src[value], 0)`
3. `XLOOKUP(@t[key], @@src[key], @@src[value], 0)`
4. 文本列 `CONCAT(@t[prefix], '_', @t[suffix])`
5. 跨表标量引用 `@other_table[col] + @this[col]`

## 七、性能基线采集与解读

每次执行 `execute_formula_on_column` 与 `recalculate_downstream_dag` 都会向
`_perf_log` 写一条记录：

| 字段 | 含义 |
|------|------|
| `op` | 操作名（`execute_formula_on_column` / `recalculate_downstream_dag`） |
| `table_name` / `column_name` | 目标定位 |
| `n_rows` | 实际写回行数 |
| `elapsed_ms` | 耗时（毫秒） |
| `extra_json` | 引擎、ref_tables、rows_changed、skipped_unchanged、cached_tables（本轮 DAG 复用的窄投影缓存条目数）、cycle_skipped 等附加信息 |
| `created_at` | UTC 时间戳 |

### 推荐基线脚本

```sql
-- 最近 100 次重算耗时
SELECT op, table_name, column_name, n_rows, elapsed_ms,
       json_extract(extra_json, '$.engine') AS engine, created_at
FROM _perf_log
ORDER BY id DESC LIMIT 100;

-- 各表平均耗时与 P95
SELECT table_name, COUNT(*) AS n, AVG(elapsed_ms) AS avg_ms,
       MAX(elapsed_ms) AS max_ms
FROM _perf_log
WHERE op='execute_formula_on_column'
GROUP BY table_name
ORDER BY avg_ms DESC;

-- DAG 一次重算波及节点数
SELECT id, n_rows, elapsed_ms,
       json_extract(extra_json, '$.n_affected') AS n_affected,
       json_extract(extra_json, '$.rows_changed') AS rows_changed,
       json_extract(extra_json, '$.skipped_unchanged') AS skipped_unchanged,
       json_extract(extra_json, '$.seeds') AS seeds
FROM _perf_log
WHERE op='recalculate_downstream_dag'
ORDER BY id DESC LIMIT 20;
```

### 上线前后对比建议

1. 升级前打一份基线（同样的公式集执行 N 次 → `elapsed_ms` 中位数）。
2. 升级后跑同一组公式，比较 `elapsed_ms` 分布。
3. A 系预期改善：`load + writeback` 阶段下降明显；`call_calculator` 高频公式
   `elapsed_ms` 降幅最大。
4. B 系预期改善：单表大行数 + 简单算式公式 `engine=duckdb` 比 `pandas` 显著更快。

## 九、风险提示

- **索引名称长度**：自动复合索引采用 `idx_calc__<10位 sha1>`，远低于 SQLite 软上限。
- **`_perf_log` 体积**：长期累积建议定期归档（`DELETE FROM _perf_log WHERE created_at < '...'`），
  本次未做自动清理。
- **DuckDB 翻译白名单**：第一版只支持四则运算 + 同表引用，**不要**对包含
  `call_calculator` / `vlookup` / `cumsum_*` 的公式抱有 DuckDB 加速预期 ——
  这些会自动 fallback 到 Pandas 路径，不会出错也不会加速。
- **Provenance 含义不变**：批量回写后 `_cell_provenance.source_tag` 仍为
  `formula_computed`，与历史一致；下游审计逻辑无需调整。
