## 二、跨表公式（@@ INDEX 类型）性能诊断

### 2.1 什么是跨表公式

跨表公式指使用 `@@` 整列引用 + `INDEX` / `XLOOKUP` / `MATCH` 等函数在多个表之间做数据查询的公式。典型格式：

```
INDEX(@@other_table[col], @this_table[level]) * factor + INDEX(@@another_table[col], ...)
```

### 2.2 性能特征

| 维度 | 值 |
|------|-----|
| 单列执行时间 | 100–110ms（600行 × 5–6个外部表） |
| 涉及项目4的列数 | ~60 列（约占 290 条公式的 20%） |
| 最重表 | `player_total_attr`（15 列，全部跨表） |
| 执行引擎 | Pandas（eval_series → safe_eval_vector） |
| DuckDB 支持 | **否** — 被 `@@` 白名单排除 |

### 2.3 热点分布

经过 AST 预编译优化后，单列 `player_total_attr.hp` 的耗时分解：

| 阶段 | 耗时 | 占比 |
|------|------|------|
| 常量处理 + 解析引用 | <1ms | ~1% |
| 加载主表 (600行 × 3列) | 1.7ms | ~1.5% |
| 加载 6 个外部表 (各 1列) | 0.3–0.5ms × 6 | ~3% |
| **公式计算 (eval_series)** | **105ms** | **~95%** |
| 写入 + 来源追踪 | <1ms | ~1% |

瓶颈：**Pandas eval_series 逐行 INDEX 查找**。

### 2.4 当前公式分类

项目4的 290 条公式按计算引擎分类：

| 引擎 | 数量 | 示例表 | 速度 |
|------|------|--------|------|
| DuckDB | ~120 | `num_base_framework.*` (简单四则) | 2ms/列 |
| Pandas 轻量 | ~110 | `equip_base.*` (row型, 带cross_refs跳过) | <10ms/列 |
| **Pandas 跨表** | **~60** | **`player_total_attr.*`** | **110ms/列** |

## 三、DuckDB 扩展方案

### 3.1 当前 DuckDB 白名单限制

`duckdb_compute.py` 的 `_check_whitelist` 当前拒绝：
- `@@table[col]` — 整列引用
- 跨表 `@other_table[col]`
- `INDEX` / `XLOOKUP` / `MATCH` / `VLOOKUP` 等查找函数
- `call_calculator` / `CUMSUM_*` / `interp` / `piecewise` 等高级函数
- `&` / `|` 位运算/拼接

### 3.2 扩展路线

#### 阶段 B2（推荐优先）

**目标**：支持 `@@` 整列引用 + `INDEX` + 跨表 `@` 引用

**技术方案**：

1. **跨表数据传递**：对每个 `@@table[col]` 引用，加载整列数据到 DuckDB 内存表中
2. **DuckDB SQL 翻译**：
   - `@@table[col]` → 导入为 DuckDB 表 `__tbl_table__col`
   - `@table[col]` → JOIN 对应表
   - `INDEX(arr, idx)` → `arr[idx]`（DuckDB list 索引）
3. **SQL 生成**：`_check_whitelist` 扩展白名单，`compute_column_via_duckdb` 支持多表 JOIN

**预期收益**：
- `player_total_attr` 单列从 110ms → <10ms（10×+ 提速）
- 端到端常量修改：228ms → <50ms

**风险**：
- DuckDB 内存表创建/销毁有固定开销（~1-2ms），对小表可能不如 Pandas
- INDEX 语法翻译需要处理边界（越界、空值）
- 需要适配 `level_range` 过滤

#### 阶段 B3（可选）

**目标**：支持 `call_calculator` 在 DuckDB 路径中

**方案**：calculator 结果预先计算为 DuckDB 表，JOIN 到主查询

#### 阶段 B4（长期）

**目标**：Pandas 路径全面退役，所有公式统一走 DuckDB

### 3.3 白名单扩展伪代码

```python
# 扩展 _ALLOWED_FUNCS
_ALLOWED_FUNCS = {"min", "max", "abs", "round", "sqrt", "const_value", "if", "index"}

# _check_whitelist 中：
# 1. 允许 @@ 引用（记录需要导入的外部表+列）
# 2. 允许跨表 @ 引用（转为 DuckDB JOIN）
# 3. 将 INDEX(arr, idx) 翻译为 arr[idx + 1]（DuckDB list 是 1-indexed）

def _check_whitelist_v2(formula, table_name, columns_in_table):
    cross_refs = {}  # @@table[col] → DuckDB 表名
    for tbl, col in _AREF.findall(formula):
        duck_name = f"__tbl_{tbl}__{col}"
        cross_refs[(tbl, col)] = duck_name
    
    for tbl, col in _REF.findall(formula):
        if tbl != table_name:
            duck_name = f"__tbl_{tbl}__{col}"
            cross_refs[(tbl, col)] = duck_name
    
    ...
    return used_cols, sql_expr, cross_refs
```

## 四、当前架构可观测性

### 4.1 性能日志

`_perf_log` 表记录每次公式执行的关键指标：

```sql
SELECT op, table_name, column_name, n_rows, 
       ROUND(elapsed_ms,1) as ms, 
       json_extract(extra_json, '$.engine') as engine
FROM _perf_log ORDER BY id DESC;
```

引擎字段：`duckdb` / `pandas`（空=旧路径）

### 4.2 性能开关

通过 `project_settings` 表的 `perf` 键控制：

```json
{
  "use_min_column_load": true,
  "use_batch_writeback": true,
  "use_batch_lookup": true,
  "use_dag_recalc": true,
  "use_duckdb_compute": true
}
```

所有开关可通过 API 或在数据库中修改，无需重启服务。
