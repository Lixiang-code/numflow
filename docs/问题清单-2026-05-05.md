### P0-1：Agent Session 客户端断连后永久卡 `running`

**源文件**：`app/routers/agent.py:325-327`

**当前代码**：
```python

    pass
```

**问题**：用户刷新页面 / 关闭标签 / 网络断连后，`StreamingResponse` 的 generator 收到 `GeneratorExit`，该异常被静默吞掉，session 永远保持 `status='running'` + `finished_at=NULL`。后续 `get_resumable_session()` 会将其误判为可恢复而重复捞起。

**卡死案例**：Session #42 (`gameplay_table.monster_final`)，2026-05-05 07:35 CST 开始，已卡死超过 12 小时。

**修复方案**（`agent.py` 共 2 处改动）：

**改动 1** — GeneratorExit 路径补充终止标记（替换第 325-327 行）：
```python
except GeneratorExit:
    try:
        update_agent_session(
            conn, session_id,
            status="error",
            error_text="client_disconnected",
            design_text="".join(design_buf),
            review_text="".join(review_buf),
            execute_text="".join(execute_buf),
            tools_json=json.dumps(tools, ensure_ascii=False),
            events_json=json.dumps(tracked_events, ensure_ascii=False),

            current_phase=_current_phase,
            completed_phases=json.dumps(_completed_phases),
            finished=True,
        )
    except Exception:
        pass
```

**改动 2** — `_session_tracking_wrapper` 的 `try:` 块增加通用异常兜底（当前只 catch 了 `GeneratorExit`，生成器内部其他未捕获异常同样导致 session 卡 running）。在 `except GeneratorExit:` 之后新增：

### P0-2：`list_exposed_params` 工具始终返回 JSONDecodeError

**源文件**：`app/services/agent_tools.py:3861-3865`

**当前代码**：
```python
def _trim_value(v: Any) -> Any:
    s = json.dumps(v, ensure_ascii=False)
    if len(s) > 200:
        return json.loads(s[:200] + '..."')  # ← 危险
    return v
```

**根因**：当 value_json 序列化后超过 200 字符时，对前 200 字符做硬截断 `s[:200]` 然后追加 `..."` 试图补全。但截断点可能恰好落在 JSON 转义序列中间（如原始字符串含 `\n`，经 `json.dumps` 后变成 `\\n`，截断在 `\\` 之后），形成非法 JSON 导致 `json.loads()` 抛出 `JSONDecodeError: Invalid \escape`。

此函数未被 try/except 包裹，异常直接传播到 `dispatch_tool`。

**影响范围**：`_list_exposed_params`（第 3874 行调用）以及任何使用 `_trim_value` 的地方。Agent 无法读取上游暴露参数，被迫"盲写"设计。这是项目 5 的 27 条反馈中占比最高的问题（18/27 = 67%），同时也是全场 64 条反馈中唯一重复超过 5 次的问题。

**修复方案**（第 3864 行）：
```python
def _trim_value(v: Any) -> Any:
    s = json.dumps(v, ensure_ascii=False)
    if len(s) > 200:
        try:
            return json.loads(s[:200] + '..."')
        except (json.JSONDecodeError, ValueError):
            return s[:197] + '...'
    return v
```

## 四、P1 高优先级（共 4 项）

### P1-1：CUMSUM_TO_HERE 不支持 3D 表 dim1 分组累加

> 反馈：项目5 #13, #15 | 项目3 #15, #16, #17 — 共 5 次

**源文件**：`app/services/formula_engine.py:389-393`

**当前代码**：
```python
def _cumsum_to_here(arr: Any) -> float:
    i = _CURRENT_ROW_INDEX.get()
    vals = _arr_to_floats(arr)
    return float(sum(vals[: i + 1]))
```

**问题**：`_CURRENT_ROW_INDEX` 是全局行号，不感知 3D 表的 dim1 边界。当 dim1 分组切换时（如 artifact_id 从 1 变到 2），累计值不会重置，导致分组 2 的累计值错误地包含了分组 1 的数值。

**修复方案**（推荐新增函数）— `formula_engine.py` 新增：
```python
def _cumsum_group_by(dim_col: Any, arr: Any) -> float:
    """CUMSUM_GROUP_BY(@@dim_col, @@arr): 按 dim_col 值分组后组内累加"""
    i = _CURRENT_ROW_INDEX.get()
    dim_vals = _arr_to_floats(dim_col)
    arr_vals = _arr_to_floats(arr)
    current_dim = dim_vals[i]
    total = 0.0
    for j in range(i + 1):
        if dim_vals[j] == current_dim:
            total += arr_vals[j]
    return total
```

并在 `_FUNCTIONS` 字典（第 494-495 行附近）注册：
```python
"cumsum_group_by": _cumsum_group_by,
```

同时在 `agent_tools.py` 的工具描述（`CUMSUM_TO_HERE` 相关部分）增加新函数的使用说明。
