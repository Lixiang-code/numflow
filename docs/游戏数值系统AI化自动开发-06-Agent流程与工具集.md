# 06 — Agent 流程与工具集

> 与 `05` 配合：本节为 **角色、流程、提示词原则** 与 **工具名级规范**；工具与 FastAPI 路由的对应关系见 `04`。  
> 执行数值流水线时仍以 `03` 为准。

## Agent 流程设计与提示词设计原则

### Agent 的角色划分

整个系统中的 Agent 并非单一角色，而是根据任务阶段划分为两类。**初始化 Agent** 负责项目首次构建，接收用户填写的项目配置，推导需要建立哪些表、每张表的结构是什么、表间依赖关系如何，完成建表、初始填充、README 生成的全流程。**维护 Agent** 负责项目建立后的所有增量操作，响应用户的变更请求，执行局部更新、重算、验证和文档维护。两类 Agent 共享同一套工具集，但系统提示词的职责描述和工作重心不同。

---

### 初始化 Agent 流程

初始化流程分为五个阶段，Agent 应当严格按顺序执行，每个阶段完成后在内部记录阶段状态再进入下一阶段，不允许跨阶段跳跃。

**第一阶段是理解项目配置。** Agent 读取全部固定层配置，包括核心定义、游戏系统开关、属性系统定义，并通读全局 README。在此阶段 Agent 不执行任何写操作，只做理解和规划，并在内部输出一份"待建表清单草稿"，列明每张表的名称、用途、列结构、依赖上游。

**第二阶段是规划确认。** 将待建表清单草稿以结构化方式呈现给用户，说明每张表的建立理由，等待用户确认或修改。用户确认后规划冻结，后续阶段不再改变表结构，除非用户主动发起新的变更请求。

**第三阶段是建表与 README 初始化。** 按照依赖拓扑顺序依次建表，每张表建立后立即生成对应的 README，写明表的用途、列定义、依赖关系和当前状态（空表待填充）。

**第四阶段是数据填充。** 按照依赖顺序从上游表到下游表依次填充，优先填充可由配置直接推导的静态数据，再填充需要公式计算的派生列，最后调用算法 API 填充无法表格化的计算结果。每列填充完成后更新来源标记。

**第五阶段是验证与文档收尾。** 运行内置验证规则，对关键等级区间计算平衡指标，将验证结果写入全局 README，并更新所有受影响表的 README 为"已完成初始化"状态。

---

### 维护 Agent 流程

维护 Agent 的每次执行遵循"理解—定界—执行—验证—记录"五步循环。

收到变更请求后，Agent 首先读取变更涉及的所有表的 README 和相关上下游依赖关系，在内部建立本次变更的影响范围图，明确哪些表需要重算、哪些算法 API 需要重新调用。然后检查保护列表，将用户手动编辑的单元格从影响范围中排除，如果保护列表与重算范围存在冲突，需在执行前向用户说明冲突并请求确认。执行阶段按拓扑顺序推进，每一步写入后附带来源标记。执行完成后运行验证规则，将结果与上次验证结果做比对，若出现关键指标的显著偏移需主动向用户提示。最后更新所有受影响表的 README，记录本次变更内容、变更时间、和变更影响范围。

---

### 提示词结构设计原则

**系统提示词分为四个固定区块。**

第一区块是角色与职责定义，明确说明 Agent 当前是初始化角色还是维护角色，核心职责是什么，以及最重要的行为约束——不覆盖用户手动数据、不跨阶段跳跃、不在未获确认的情况下执行破坏性写操作。

第二区块是项目上下文注入，每次调用时动态注入当前项目的全局 README、核心定义配置摘要、以及当前任务直接相关的表的 README。这个区块的内容随任务不同而变化，不应当一次性把所有表的 README 全部塞入，而是由 Agent 先读取依赖图，按需拉取相关表的文档。

第三区块是工具使用规范，说明每类工具的调用时机和约束。读取工具在规划阶段自由使用；写入工具在执行阶段按拓扑顺序使用，每次写入必须附带来源标记；算法 API 工具在调用前必须确认参数含义与 API 元描述匹配，不允许用猜测的参数调用算法 API。

第四区块是输出格式规范，定义 Agent 在不同情况下应当输出什么。规划阶段输出结构化的待建表清单或影响范围图；执行阶段输出每步操作的简要日志；验证阶段输出对比报告；遇到不确定情况时输出明确的问题描述而非自行假设继续执行。

---

### 提示词设计的核心原则

**上下文按需注入而非全量注入。** 数据库内容会随项目规模增长而变得庞大，每次调用都注入全量数据会严重压缩 Agent 的有效推理空间。正确做法是 Agent 先通过读取工具获取依赖图和 README，自行判断需要哪些表的详细数据，再按需拉取。系统提示词本身保持精简，任务相关的具体数据通过工具调用动态获取。

**推理过程要求显式化。** 提示词中明确要求 Agent 在执行任何写操作之前，先在内部输出影响范围分析和执行计划，再开始执行。这不只是为了调试方便，更是为了让 Agent 在规划阶段就发现潜在的冲突或遗漏，而不是执行到一半才报错。

**不确定性处理要明确规定。** 提示词需要明确告诉 Agent 什么情况下应当停下来问用户，而不是自行假设继续执行。基本原则是：参数含义不确定时问、影响范围超出预期时问、验证结果出现显著偏移时问。但对于有明确规范可依据的推导性工作，Agent 应当自主完成，不应频繁打断用户确认。这个边界需要在提示词中用具体例子描述清楚，而不是只给一个抽象原则。

**README 是提示词的延伸。** 每张表的 README 本质上也是提示词的一部分，Agent 读取 README 的过程等同于接收关于这张表的专项系统提示。因此 README 的写作风格应当对齐系统提示词的风格——精准、无歧义、包含足够的约束说明。Agent 在更新 README 时，应当以"下一个读取这份文档的 Agent 能否仅凭这份文档理解该表的完整上下文"作为质量标准。

## 工具集设计

### 设计原则

工具集的设计目标是让 Agent 能够通过组合调用完成所有任务，而不依赖任何硬编码的业务逻辑。每个工具职责单一、边界清晰，业务逻辑由 Agent 在推理层组合实现，工具本身只负责忠实执行。每个工具都需要有机器可读的元描述，包括工具名称、用途说明、参数定义、返回值结构、以及典型调用场景，这是 Agent 能够自主选择正确工具的前提。

工具整体分为四类：**元信息工具、数据读写工具、公式与计算工具、验证与分析工具。**

---

### 元信息工具

元信息工具负责 Agent 对项目结构的认知建立，是所有其他工具调用的前提。

`get_project_config` 返回当前项目的全局配置，包括核心定义、游戏系统开关、属性系统定义的完整内容，以及全局 README 文本。Agent 在任何任务的起点都应当首先调用此工具建立基础上下文。

`get_table_list` 返回当前数据库中所有表的名称、类型标签（固定层或动态层）、以及每张表的简短用途描述。Agent 在不确定当前项目有哪些表时调用此工具，而不是依赖记忆。

`get_table_readme(table_name)` 返回指定表的完整 README 文本，包括用途说明、列定义、依赖关系和当前维护状态。Agent 在读取或写入任何表之前应当先调用此工具。

`get_dependency_graph(table_name, direction)` 返回指定表的依赖关系图，`direction` 参数支持 `upstream`（该表依赖哪些上游）、`downstream`（哪些表依赖该表）、`full`（完整子图）三种模式。Agent 在规划更新范围时调用此工具，而不是自行推断依赖关系。

`update_table_readme(table_name, content)` 写入或更新指定表的 README 文本。Agent 在完成对某张表的任何写操作之后都应当调用此工具更新文档状态。

`update_global_readme(content)` 更新全局 README，通常在初始化完成或重大变更完成后调用。

---

### 数据读写工具

`read_table(table_name, columns, filters, level_range)` 读取指定表的数据，支持按列名筛选、按条件过滤、按等级区间切片。返回结构化的数据对象，包含数据内容和每列的来源标记统计。Agent 应当按需切片而非每次读取全表。

`read_cell(table_name, row_id, column_name)` 读取单个单元格的值及其来源标记（用户手动输入 / AI 生成 / 算法派生 / 公式计算）。用于 Agent 在修改前确认该单元格是否处于保护状态。

`get_protected_cells(table_name)` 返回指定表中所有标记为"用户手动输入"的单元格坐标列表。Agent 在执行任何批量写操作之前必须调用此工具，将保护单元格从写入范围中排除。

`write_cells(table_name, data, source_tag)` 批量写入单元格数据，`data` 为列名到值列表的映射，`source_tag` 必须显式指定为 `ai_generated` 或 `algorithm_derived`，工具层拒绝接收空来源标记的写入请求。写入时自动检查保护列表，若目标单元格存在保护标记则跳过并在返回结果中列出被跳过的单元格。

`create_table(table_name, schema, readme_content)` 创建新表，`schema` 定义列名、数据类型、是否允许公式、依赖上游，同时要求传入初始 README 内容，建表和文档初始化作为原子操作不可分离。

`delete_table(table_name, confirm)` 删除表，`confirm` 参数必须显式传入 `true` 才执行，防止 Agent 因误判而触发破坏性操作。工具执行前检查是否有其他表依赖该表，若存在下游依赖则拒绝执行并返回依赖列表。

---

### 公式与计算工具

`register_formula(table_name, column_name, formula_string)` 为指定列注册公式字符串，公式中使用 `@表名[列名]` 语法引用其他表的列数据。工具在注册时执行语法检查和依赖提取，将引用关系自动写入依赖图，并对公式涉及的完整依赖链做循环检测，存在循环依赖时拒绝注册并返回冲突路径。

`execute_formula(table_name, column_name, level_range)` 执行指定列的已注册公式，在受控命名空间中通过 Pandas 计算，结果写回对应列并标记来源为 `formula_computed`。支持 `level_range` 参数做局部重算，不必每次全列重算。

`call_algorithm_api(api_name, params)` 调用算法层的指定 API，`api_name` 对应算法 API 注册表中的名称，`params` 为参数字典。工具在调用前对照 API 元描述做参数名称和类型校验，不匹配时拒绝执行并返回元描述供 Agent 参考。调用结果以结构化格式返回，Agent 决定是否写回数据库以及写入哪张表的哪一列。

`get_algorithm_api_list()` 返回所有已注册算法 API 的名称、用途说明、参数定义和返回值结构。Agent 在不确定某个计算场景应调用哪个 API 时调用此工具，而不是凭记忆猜测 API 名称。

`recalculate_downstream(table_name, column_name)` 以指定表的指定列为起点，沿依赖图向下游传播重算，按拓扑顺序依次触发所有下游公式列的重算。Agent 在修改上游数据后调用此工具，而不是手动枚举下游表逐一重算。

---

### 验证与分析工具

`run_validation(table_name)` 对指定表运行内置验证规则，返回验证报告，包括每条规则的通过状态、不通过的具体行列位置、以及问题描述。验证规则本身存储在表的元数据中，由初始化阶段写入，用户和 Agent 均可通过更新元数据新增或修改验证规则。

`run_balance_check(level_range, metrics)` 计算指定等级区间的平衡指标，`metrics` 支持指定计算哪些指标，例如自击杀回合数、伤害期望区间、属性成长斜率等。返回指标值列表以及与上次检查结果的对比差异。Agent 在每次重大变更后应主动调用此工具，而不是等用户要求。

`compare_snapshot(snapshot_id)` 将当前数据状态与指定快照对比，返回发生变化的表、列和值范围。工具层在每次初始化完成和每次重大变更完成时自动创建快照，Agent 可通过此工具向用户提供清晰的变更前后对比说明。

`get_validation_history(table_name)` 返回指定表的历史验证记录，包括每次验证的时间、触发原因和结果摘要。Agent 在判断某张表当前状态是否可信时可参考此工具，而不是只依赖最新一次验证结果。

---

### 工具层的统一约束

所有工具的返回值遵循统一结构：`status`（success / error / partial）、`data`（实际返回内容）、`warnings`（非致命性问题列表）、`blocked_cells`（被保护列表跳过的单元格，仅写入类工具返回）。Agent 在每次工具调用后必须检查 `status` 和 `warnings`，不允许忽略 `partial` 状态直接继续执行后续步骤。

写入类工具全部设计为幂等操作，相同参数多次调用的结果一致，Agent 在网络异常或执行中断后可安全重试，不会产生重复数据或状态不一致。
