## 七、发现的问题与已修项

### 本轮已修

#### Bug 1：`dispatch_tool` 未处理 `JSONDecodeError`（第 1 轮发现，已修）

**位置**：`backend/app/services/agent_tools.py` `dispatch_tool()` 第 623 行  
**现象**：qwen3.6-plus 在 execute 阶段偶发返回非 JSON 格式的 tool call arguments，导致 `json.loads()` 抛出未捕获的 `JSONDecodeError`，uvicorn ASGI 层报 500，SSE 流被强制中断（client 端 `ChunkedEncodingError(ProtocolError('Response ended prematurely'))`）。  
**修复**：在 `json.loads(arguments)` 外加 `try/except json.JSONDecodeError`，失败时返回 `{"status": "error", "warnings": ["arguments JSON decode failed"], ...}` 结构，让 execute 循环能感知错误并继续。

#### Bug 2：恶化的 tool_call 历史污染下一轮 DashScope 请求（第 2 轮发现，已修）

**位置**：`backend/app/services/agent_runner.py` execute 循环  
**现象**：qwen3.6-plus 返回 tool_call 时 `function.arguments` 偶含非 JSON 字符串。将其原样追加到 `execute_messages`，下一轮调用 DashScope 时被拒：`400 InternalError.Algo.InvalidParameter: The "function.arguments" parameter must be in JSON format`。  
**修复**：追加到 `execute_messages` 前用 `_safe_args()` 函数校验并将非法 JSON sanitize 为 `"{}"`，避免消息历史污染。

#### Bug 3：SSE 连接异常时未保留已接收事件（第 1 轮发现，已修）

**位置**：`scripts/regression_run.py` `sse_chat()` 函数  
**现象**：使用 `requests.iter_lines()` 时，若连接中途异常（如服务端崩溃），异常会向上传播并丢失已接收的 SSE 事件，导致测试分析完全空白（design3=False 等）。  
**修复**：将内层 `for raw_line in resp.iter_lines()` 包裹在 `try/except` 中，异常时记录 `conn_error` 但返回已收到的 `events` 列表。

---

### 遗留问题

#### Issue 1：execute max_rounds=8 不足以支持完整任务 ⚠️

**现象**：两个用例均超过 max_rounds（R1: 10 次，R2: 9 次），导致 execute 阶段以错误事件结束，而非正常 done 事件。  
**分析**：
- R2（init 建表）：创建表 + 读取确认 + 分批写入多行 + 更新 README 等顺序操作，天然需要 6-10 次工具调用，超出 8 轮合理。建议对 init 模式单独配置 max_rounds=12~16 或拆分为多对话轮次。
- R1（maintain 元查询）：模型将「用一句话说明你能做什么」解读为「建表任务」，这是提示词未明确区分「元查询」与「数值策划任务」的问题。建议在 maintain 模式 system prompt 中增加对话意图判断段：**若用户询问工具能力/Agent 身份，直接用自然语言回答，不调用工具**。

#### Issue 2：R1 prompt_route hit=False 导致路由提示词为 LLM 临时生成

**现象**：R1 "用一句话说明你能做什么" 触发路由器返回 hit=False（符合预期），但临时生成的 prompt 仍然指向建表任务（步骤 2 基础属性框架），进一步强化了模型对元查询的误解。  
**建议**：在 prompt_router.py 增加元查询检测逻辑，若检测到元查询（询问 Agent 身份/能力/流程），直接返回固定短提示词：「这是关于 Agent 自身能力的元查询，请直接用自然语言回答，无需调用工具。」

## 八、下次迭代建议

1. **增加 init 模式 max_rounds**：将 `max_rounds` 在 init 模式下提升至 12~16，或支持通过 `ChatBody` 参数自定义。
2. **元查询检测**：在 maintain 模式 system prompt 及 prompt_router 中增加元查询豁免逻辑，避免模型对简单询问滥用工具。
3. **execute 阶段流式输出**：当前 execute 阶段使用非流式 LLM 调用（无 `stream=True`），每轮等待时间长（可达 1-2 分钟），建议切换为流式以提升用户体验，并允许测试更早感知 done 事件。
4. **SSE iter_lines read_timeout 调整**：对于 execute 非流式调用，两次工具结果之间可能出现长时间沉默（模型思考），建议将 `read_timeout` 进一步提升至 900s 或切换为流式。
5. **blocked_cells 监控**：本轮两个用例 blocked_cells=0，说明保护格机制未被触发；建议增加一个专项用例验证 user_manual 保护逻辑。
6. **R1 任务描述优化**：将 R1 maintain 用例改为更明确的增量维护任务（如「为基础属性表补充一列暴击伤害，值=暴击率×2」），避免元查询语义歧义。
