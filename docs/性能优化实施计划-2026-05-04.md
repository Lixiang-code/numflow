# NumFlow 性能优化实施计划

> 生成时间：2026-05-04  
> 适用范围：`/www/wwwroot/numflow/backend`  
> 目标：解决“表全部完成后，全关联巨型表结构下计算效率明显变差”的问题  
> 总体策略：**先执行方案 A（当前架构内优化），再执行方案 B（SQLite + DuckDB 双层计算架构）**
>
> **状态更新（2026-05-05）**：A1-A5 与 B1 骨架已在主干代码落地。当前这份文档更适合作为
> “已完成能力 + 后续阶段路线图”阅读，而不是未开始的实施计划。现阶段的主要增量目标应下沉为：
> 1. B2：DuckDB 支持 `@@` / `INDEX` / 跨表引用；
> 2. 性能日志与部署文档收口；
> 3. 前端减少逐列重算请求。

---

## 一、背景与问题定义

当前系统在业务表逐步补齐、公式链变长、关联关系增多后，已经从“小规模工具式计算”进入“中大型重算场景”。  
从现有代码路径看，性能问题主要不是单点 SQL 慢，而是**执行模型在大规模表/依赖图下放大了成本**：

1. **全表加载过多**
   - `formula_exec.load_table_df()` 直接 `SELECT *` + `pd.read_sql_query`
   - 公式只依赖少数列，也会把整张表搬入 Pandas

2. **逐行回写过多**
   - `execute_formula_on_column()` / `execute_row_formula()` 每行单独 `UPDATE`
   - 同时写 `_cell_provenance`
   - 行数一大，SQLite 往返成本显著

3. **依赖链串行重算**
   - `recalculate_downstream()` 以列为粒度串行触发
   - 易出现重复扫表、重复加载、重复写回

4. **call_calculator 热路径退化为大量小查询**
   - 公式中每个 `call_calculator()` 都会走 `_calculators` 查配置 + 再查目标表
   - 在整列/整表执行时，会形成高频 lookup

5. **索引不足**
   - 当前系统表与元数据表缺乏针对高频筛选条件的复合索引
   - 如 `level/gameplay/attr/res`、依赖图方向查询、公式注册表定位等

---

## 二、性能目标

本轮优化不追求一次性“彻底重构”，而是按阶段建立可持续的性能体系。

### Phase A 目标（当前架构内）

1. 同等数据规模下，整列公式执行时间明显下降
2. `call_calculator` 驱动的整表重算不再出现“每个单元格一个小 SQL”的退化
3. 下游重算从“逐列串行”转为“按依赖批量”
4. 不改变对现有业务表/元数据表的兼容性

### Phase B 目标（计算层升级）

1. 保留 SQLite 作为在线业务存储与元数据存储
2. 引入 DuckDB 作为批量重算与大关联计算层
3. 把“重算成本”从 Python/Pandas/单条 SQL 往返迁移到向量化计算引擎

---

## 三、总体路线

| 阶段 | 策略 | 目标 | 风险 |
|------|------|------|------|
| **A1** | 索引与最小列加载 | 低风险快速止血 | 低 |
| **A2** | 批量回写与 calculator 批量 lookup | 解决 Python/SQLite 往返瓶颈 | 中低 |
| **A3** | DAG 批量重算 | 解决重复扫表/重复计算 | 中 |
| **B1** | DuckDB 验证性接入 | 建立可行的高性能计算层 | 中 |
| **B2** | DuckDB 批量重算落地 | 将大关联计算迁到向量化执行 | 中高 |

> 注：截至当前代码状态，A1-A5 与 B1 已实现；真正待做的核心工作已收缩到 B2 及其配套灰度。

## 四、方案 A：当前架构内优化

### A1. 最小列加载（高优先级）

#### 当前问题
- `load_table_df()` 一律 `SELECT *`
- 大表中很多列与当前公式无关，造成无效 IO、无效内存复制

#### 优化方案
- 为公式执行引入“**按依赖列裁剪**”机制：
  - 基于 `parse_formula_refs()` / `parse_row_refs()` / `parse_constant_refs()`
  - 只查询：
    - `row_id`
    - 目标列
    - 当前公式直接依赖的列
    - `level_column`（若有区间执行）
- 为 `load_table_df()` 增加列白名单参数，而不是无条件全表加载

#### 预期收益
- 大表公式执行的 DataFrame 构建时间显著下降
- 内存峰值下降

---

### A2. 批量回写替代逐行 UPDATE（高优先级）

#### 当前问题
- 每行 `UPDATE table SET col=? WHERE row_id=?`
- 行数高时，SQLite 调用与事务内往返成本大

#### 优化方案
- 改为 `executemany()` 批量提交
- 若收益仍不足，再升级为：
  - 临时表写入结果集
  - 一次性 merge/update 回主表
- provenance 更新也同步改为批量写

#### 预期收益
- 公式执行写回阶段提速明显
- 对“几千行 × 多列”场景收益最大

---

### A3. 给高频路径补索引（最高性价比）

#### 建议索引

1. 元数据表
   - `_dependency_graph(to_table, to_column)`
   - `_dependency_graph(from_table, from_column)`
   - `_formula_registry(table_name, column_name)`
   - `_cell_provenance(table_name, row_id, column_name)`（主键已覆盖，但需确认高频聚合是否还要扩展）
   - `_calculators(name)`（主键已覆盖）

2. 业务表 / lookup 表
   - 针对 calculator 注册时的 `axes_json` 自动建复合索引
   - 典型示例：
     - `(level, gameplay, attr)`
     - `(level, gameplay, res)`
     - `(gameplay, attr)`
     - `(gameplay, res, grain)`

#### 实施方式
- 在 `register_calculator()` 或相关 migration 中自动补索引
- 避免要求人工维护

#### 风险提示
- 索引太多会拖慢写入；因此只给高频 lookup 表与高频元数据路径建索引

---

### A4. 将 call_calculator 从“逐单元格 SQL”改为批量 lookup（高优先级）

#### 当前问题
- 公式整列执行时，`call_calculator()` 可能对每个单元格都触发一次独立查询

#### 优化方案
- 在整列公式执行开始前，预解析本次会用到的 calculator
- 对 lookup 型 calculator：
  - 预先加载为内存字典（key -> value）
  - 或构造成 DataFrame / join 中间表
- 对重复参数模式启用本地缓存
- 优先解决：
  - 相同 calculator 被整列反复调用的场景
  - 相同 `level/gameplay/attr` 组合高频重复出现的场景

#### 预期收益
- 减少成百上千次小 SQL
- 把热点从数据库调用转移到内存查找

---

### A5. DAG 批量重算（中高优先级）

#### 当前问题
- `recalculate_downstream()` 当前是“查到谁就逐个执行”
- 下游链深时容易重复计算

#### 优化方案
- 基于 `_dependency_graph` 构建列级 DAG
- 当某列变化时：
  1. 找出全部受影响列
  2. 拓扑排序
  3. 去重后按层批量执行
- 未来可扩展为：
  - 仅重算脏区间
  - 同层并行（后续若引入多进程/外部执行器）

#### 预期收益
- 避免重复扫表
- 为 DuckDB 计算层接入打基础

## 五、方案 B：SQLite + DuckDB 双层架构

### B1. 为什么是 DuckDB

相较于继续堆 Pandas，DuckDB 更适合当前场景：

1. **嵌入式**
   - 不需要引入独立数据库服务
   - 部署心智接近 SQLite

2. **强 join / 强聚合 / 强批量计算**
   - 对大表关联、过滤、分析型计算更强

3. **向量化执行**
   - 适合整列重算、批量重算

4. **与 Python/Arrow/Pandas 兼容**
   - 可逐步接入，不必推倒重来

### B2. 推荐接入方式

#### 不推荐
- 直接全量把业务库从 SQLite 换成 DuckDB

#### 推荐
- **SQLite 继续保留**
  - 业务表
  - 元数据表
  - 公式注册表
  - 依赖图

- **DuckDB 负责批量计算**
  1. 从 SQLite 拉取参与本轮重算的最小表集
  2. 在 DuckDB 中完成 join / 批量公式计算
  3. 把结果列回写 SQLite

这样做的好处：
- 不影响现有 API 与表模型
- 计算层可独立替换与演进
- 回滚风险低

## 六、推荐实施顺序

### 第一阶段：立刻落地（1 个迭代）

1. 补高频索引
2. `SELECT *` → 最小列加载
3. 逐行 UPDATE → 批量回写

### 第二阶段：结构性止血（1 个迭代）

4. `call_calculator` 批量 lookup / 缓存
5. DAG 批量重算

### 第三阶段：性能升级（1~2 个迭代）

6. DuckDB spike（验证性原型）
7. 选一条典型重算链切到 DuckDB
8. 建立“SQLite 存储 + DuckDB 计算”正式路径

## 八、需要新增的工程能力

### 1. 性能基线与观测

必须先有基线，否则很难判断优化是否真正有效。

建议记录：
- 单列公式执行耗时
- 单次重算总耗时
- 每轮加载表数 / 行数 / 列数
- `call_calculator` 调用次数
- 每轮写回行数

### 2. 压测样例

建议固定 2~3 组数据规模：
- 小型：开发自测
- 中型：常规生产体量
- 大型：接近当前“明显变慢”的真实规模

### 3. 回退开关

尤其在引入 DuckDB 时，应保留：
- `use_duckdb_compute = false/true`
- `use_batch_lookup = false/true`
- `use_dag_recalc = false/true`

## 九、风险与取舍

| 风险 | 说明 | 应对 |
|------|------|------|
| 行为偏差 | 批量执行与原逐行执行可能出现边界差异 | 增加对照回归测试 |
| 索引过多 | 写入性能可能下降 | 只对高频路径建索引 |
| DuckDB 接入复杂度 | 计算层与存储层双栈 | 先做 spike，再逐步迁移 |
| 公式引擎兼容性 | 部分函数/语义在 SQL 化时难直接映射 | 保留 Python fallback 路径 |

---

## 十、最终建议

**最终执行策略：A + B，但必须先 A 后 B。**

- **A 是必做前置**：先把当前架构内最明显的浪费去掉
- **B 是推荐升级**：当 A 把边界理顺后，再接入 DuckDB 才最稳

如果只给一句结论：

> **短期先做“索引 + 最小列加载 + 批量回写 + 批量 lookup + DAG 重算”，中期演进到“SQLite 存储 + DuckDB 计算”的双层架构。**
