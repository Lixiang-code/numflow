## 二、设计评价

### 优点

1. **信息层级清晰**  
   `sl-input-ai`（绿色实线）vs `sl-input-dev`（灰底虚线）给"AI 可见 / 仅开发者"建立了视觉语义，认知成本低。

2. **预览置顶符合工作流**  
   开发者修改提示词时最常要验证的就是渲染结果；置顶 + max-height 33vh 既常驻可见又不抢占编辑空间。

3. **AutoTextarea 体验接近原生**  
   默认行数 = 内容行数 + 1、超 10 行启用滚动条，比固定高度输入框舒适很多，且不影响布局。

4. **草稿缓存设计合理**  
   `useRef(Map)` 避免每次击键触发全量渲染；`useState(Set)` 脏状态保证 UI 响应式更新；两者各司其职。

5. **冲突检测逻辑完整**  
   baseline ≠ server 且 draft ≠ baseline 才触发弹窗，避免"空改动也弹窗"的误报。

### 不足 / 隐患

1. **`deepEqual = JSON.stringify` 的脆弱性**  
   key 顺序不同时会误报"不等"。`modules` 数组是顺序敏感的，目前问题不大；但一旦后端返回字段顺序变化，就会产生假冲突。建议后续替换为递归比较或 `fast-deep-equal`。

2. **`promptCacheKey` 使用当前 `tab` 而非 item 自身的 tab 属性**  
   `isPromptDirty(pk)` 内部用 `tab`（当前选中 tab）拼 key，若在"工具" tab 下调用但 item 实际属于"系统" tab，会 miss。目前因为 `promptItems` 仅加载当前 tab 的数据，不会触发，但这是一个潜在 bug。

3. **缓存没有大小上限**  
   Map 会随会话积累，虽然提示词条目不多，但没有 LRU/TTL，长期使用的页面（SPA 不刷新）可能积累孤立条目。

4. **"稍后再说"没有视觉提示**  
   冲突弹窗关闭后，用户如何再次看到冲突内容？目前什么标记都没有留下，容易遗忘。

5. **新建 SKILL（id='new'）的草稿在刷新后丢失**  
   缓存在内存中，刷新即失；新建条目若未保存就刷新，内容消失且无警告。

6. **Ctrl+S 在新建条目未填标题时静默失败**  
   `canSave` 为 false 时快捷键无响应，但没有任何提示告知用户"为什么不能保存"。

7. **工具提示词的 `module_key` 路径没有可视化校验**  
   开发者输入模块内容时不知道这个模块会被注入到 JSON schema 的哪个具体字段；路径展示仅在预览中，且是纯文本，没有指示"当前修改的是哪一行"。

## 三、同一思路链条下尚未执行、但应当执行的内容

### 优先级 P0（补全现有功能完整性）

#### 3.1 "稍后再说"后补充视觉提示
- 冲突弹窗关闭后，在侧边栏该条目上加 `⚡` 图标（`sl-chip amber`），hover 时提示"存在未解决冲突，点击此条目可重新选择"
- 选中该条目时重新触发冲突弹窗（目前 `selectSkill` 已判断 `serverConflict`，需确认关闭弹窗不删除 `serverConflict`）
  > 检查 `SkillLibrary.tsx`：关闭弹窗的 "稍后再说" onClick 只调用 `setConflictSkill(null)`，**没有** 删除 `entry.serverConflict`，所以下次 selectSkill 仍会触发弹窗 ✅ — 但侧边栏缺少视觉提示。

#### 3.2 新建未保存条目刷新前警告
- 在 `beforeunload` 上注册 `window.addEventListener`，若 `dirtySkillIds.size > 0 || dirtyPromptKeys.size > 0` 则弹出浏览器离开确认弹窗
- 路由切换（React Router `useBlocker`）同理

#### 3.3 Ctrl+S 失败时提示原因
- `handleSave` 中 `canSave` 为 false 时，`showToast('error', '请先填写标题才能保存')`

### 优先级 P1（体验提升）

#### 3.4 置顶预览区折叠/展开
- 预览区右上角加折叠按钮（`⌃`/`⌄`），折叠后仅显示标题栏，展开后恢复 33vh
- 折叠状态存 `localStorage`，跨刷新记忆

#### 3.5 模块顺序拖拽
- 目前模块只能启用/禁用，无法调整顺序
- 用 `@dnd-kit/sortable` 实现拖拽排序，保存时写入后端
- 顺序字段：`PromptModule` 加 `sort_order: number`（需后端配合）

#### 3.6 新建 SKILL 草稿 localStorage 持久化
- 仅对 id='new' 的草稿写入 `localStorage`（key: `sl_draft_new_${projectId}`）
- 加载页面时检查是否有未保存的 new 草稿并恢复

#### 3.7 模块批量启用/禁用
- 顶部加"全部启用 / 全部禁用"快捷操作，提高管理多模块时的效率

### 优先级 P2（工程质量）

#### 3.8 替换 deepEqual 实现
- `npm install fast-deep-equal`，替换 `JSON.stringify` 比较，解决 key 顺序问题

#### 3.9 缓存 LRU 限制
- 给 `skillCache` / `promptCache` 加最大容量（如 50 条），超出时删除最久未访问的条目

#### 3.10 工具提示词路径可视化
- 在每个工具模块的标题旁展示路径面包屑（如 `function → parameters → sql → description`）
- 高亮当前正在编辑的模块在预览 JSON 中对应的行（需预览区改为语法高亮）

#### 3.11 提示词字符计数
- 在置顶预览区下方展示"预估 token 数"（简单实现：字符数/4），帮助开发者控制提示词长度

### 缓存结构
```typescript
// key = String(skillId) 或 'new'
skillCache.current: Map<string, SkillCacheEntry>
// key = `${tab}::${prompt_key}`
promptCache.current: Map<string, PromptCacheEntry>

type SkillCacheEntry = {
  draft:          SkillItem   // 当前编辑内容
  baseline:       SkillItem   // 上次从服务端加载的快照
  serverConflict? SkillItem   // 服务端更新但本地有草稿时暂存
}
```
