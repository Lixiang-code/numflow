# 03 — 数值模型首次构建流程

> **执行顺序约束**：下列章节须按顺序完成；每步产出文档（README / 方案说明）并持久维护。  
> 依赖知识：`01` 配置结果、`02` 默认细则。

## 数值模型构建（首次）

### 整体环境确认

- 整体数值框架需要围绕有细则描述的内容展开
- 对于细则的部分，如果用户有准确描述则执行
  - 没有则判断是否贴合某些细则项目
    - 能贴合，则视为对应细则项目
    - 无法贴合，忽视，并在对应模块的readme中明确说明哪些内容被忽略，要求用户补充清晰描述
- 用户没具体说‘要’，但是被标注为默认的项目，视为用户要
- 整理出一个全局readme文件，将用户设计的和我们推断的内容写入readme
  - 包括所有核心定义\游戏系统\属性系统
  - 全局readme是游戏的核心规则，需要始终维护，跟项目实际\用户最新需求情况对齐

### 基本属性基础框架表搭建

- 以所有被选用的属性表头，标准等级列表为表列，创建基本属性框架表
- 攻击力列将根据核心规则用户的选择拉表
- 设计防御力曲线
  - 减法公式下，默认防御力为攻击力一半
  - 乘法公式下，默认防御力跟攻击力数值1：1
- 设计常数
  - 乘法公式下的防御算法常数
  - 等级化高级属性的算法常数
  - 注意常数应当优先是跟等级做运算的，除非整体膨胀平缓
- 设计高级属性曲线
  - 对于暴击、闪避等带感的属性，同时也具有容纳上限的，设计从无感->感知加强->稳定的曲线
  - 对于感知弱并且无容纳上限的，设计始终膨胀的曲线
  - 对于对抗属性（如暴击跟抗暴击），应当始终按照攻方有利设计，进攻属性要大于防御属性投放
    - 感知属性应该始终按照对抗后实际值来认定感知度
- 设计技能强度曲线
  - 需要分别设计杀伤性技能系数和生存性技能系数曲线
  - 一般来说技能效果系数应该从至少120%开始并始终膨胀，技能才能带感
    - 需要考虑游戏设计目标来衡量
- 最后确定生命值曲线
  - 一般来说以‘自击杀’跟‘战斗节奏设计’做平衡，综合所有设计的属性值，即可计算出最终生命曲线
  - 若有明确要求生命值跟攻击的数量关系，或者生命值跟攻击的比例过于扭曲，则需要考虑重新设计防御属性投放（减法公式）或者防御公司常数（乘法公式）来二次修正
    - 生命值小于攻击力，以及生命值大于攻击力的1000倍的情况，都会认为过于扭曲
  - 最终验证点：在等级曲线中生命值应该是单调递增的，不能出现凹点
- 反复检查和校验，确认最终设计合理，即完成属性框架标搭建
- 完成后，需要撰写属性基础框架表的readme，说明设计情况
- 这一步还要设计等级时间表，定义每个等级玩家会停留的时间长度（默认单位：分钟）
  - 注意设计的合理性，要考虑前期的快速进展期和平台期
- 设计经验表，定义每个顶几下的单位时间经验产出（一般需要膨胀），然后根据设计的停留时间长度，计算出每个等级的升级经验值

### 玩法系统属性方案设计

- 为每个需要落地的玩法系统（如装备，坐骑等等）做属性方案设计
- 定义每个系统需要投放的属性类型
- 基于常见的合理的游戏理解来设计
  - 若用户有明确的设计要求，以用户为准
- 注意不要有遗漏玩法或者遗漏属性，所有属性都应该有至少1个落脚系统
  - 大部分属性应该有多个系统落脚点
- 设计方案文档化，持久保存和维护

### 玩法系统属性分配表

- 为每个系统\每个属性搭建属性分配表
- 分配值是一个百分比
  - 例子：装备系统被分配到30%的生命，30%的攻击，30%的防御投放
- 分配表是按照等级下拉的，原则上占比是稳定常数，但也允许随等级膨胀，以应对更复杂的需求
- 在页面功能上，允许单独查看某个具体玩法的属性分配表，也允许单独查看某个属性的玩法分配情况
- **不要求分配结果求和时=100%**
  - 完全允许求和大于100%和小于100%的情况
  - 不同等级属性系统启动情况不同，并且后期仍然会膨胀，所以需要允许更自由的分配方式
- 需要维护一个readme说明设计情况

### 基本属性二阶框架表

- 在玩法属性分配后，由于属性的实际比例可能不再是100%，需要搭建二阶框架表
- 二阶框架标中，每个属性的值=基础框架表值*玩法系统分配表分配比加总
- 构建完成后，验证二阶框架表中最终战斗节奏是否平衡
  - 最重要的就是防御平衡，若有显著的失衡，重新设计防御常数或者防御投放比例
  - 迭代计算并再次验证，直到二阶框架表展示出来的数值效果合理

### 玩法系统属性表

- 以框架表+玩法分配表，制作出玩法的具体属性表
- 这里可以知道每个玩法在每个标准等级下获得的具体属性情况

### 养成资源设计

- 基于玩法系统，设计处需要的养成资源类型
- 至少应该包含2总基础货币消耗
- 原则上系统之间出力基础货币可以复用外，专属养成用品应该隔离
  - 用户有要求则以用户为准
- 本步骤需要把所有涉及的养成资源罗列出来，定义好养成资源对应哪个玩法或者哪些玩法
- 对于多玩法复用的养成资源，需要有一个基础的权重分配设计

### 养成资源基础框架表

- 设定每个养成资源在每个等级下的单位时间产出（一般以分钟计）
- 根据常见的或者有思考的设计，定义每个资源是否随着等级膨胀，膨胀的话膨胀率如何
  - 资源膨胀可以让玩家有‘越来越多’的成长感，但不是任何道具都适合
- 结合每个等级的停留时间，计算出每个等级下资源的实际产量和总产量
- 需要检查合理性

### 养成资源分配表

- 为每个资源\每个玩法\每个等级定义具体的消耗比例
- **不要求分配结果求和时=100%**
  - 完全允许求和大于100%和小于100%的情况
  - 不同等级属性系统启动情况不同，并且后期仍然会膨胀，所以需要允许更自由的分配方式

### 养成资源定量表

- 根据养成资源基础框架和分配表，可以得出每个养成玩法每个具体等级下可以使用的数量
- 玩法表可以直接@对应资源表的等级，知道该阶段可用资源量

### 玩法系统落地表

- 根据玩法系统属性表以及玩法特征，逐个落地玩法系统表
- 注意这里有大量细节设计需要完成，是个长过程
- 关于资源利用，需要注意可用资源是一个差值（目标等级总产量-前一目标等级总产量）

#### 玩法设计参考原则

- 概率平衡
  - 对于具有概率产出属性的玩法（如洗练），需要基于游戏生命周期设计一个可以转换为时间曲线的期望图标
- 消耗设计
  - 基于可用产量和培养频率，设计消耗
  - 注意这里是可以强行拉表某些凹点的
    - 建议使用权重分配方式，保证总量合理同时权重曲线合理
