# 04 — 前后端框架与部署

> **定位**：运行形态、技术选型、网络与进程、访问控制、磁盘布局。  
> **不包含**：表结构元数据、公式语法、Agent 流程（见 `05`、`06`）。  
> **与产品需求的关系**：`01` 描述业务界面与流程；本节描述 **如何实现** 该 Web 工具的工程底座。

## 整体架构选型思路

这个工具的用户规模极小（个人或小团队），但计算逻辑重、AI 调用链长、数据状态复杂，因此架构选型的核心矛盾不是并发能力，而是**开发效率与交互复杂度之间的平衡**。纯 Streamlit 方案开发最快，但随着功能增长会在交互层遭遇明显天花板——Streamlit 的状态模型与游戏数值工具需要的细粒度交互（行内编辑、实时公式预览、多面板联动）并不匹配。因此采用前后端分离架构：**FastAPI 后端 + React 前端**，SQLite 作为数据层，部署在 Linux 服务器上，用户通过浏览器访问。

---

## 后端：FastAPI

后端选用 FastAPI，核心理由是它与整个 Python 技术栈天然融合，Pandas、算法计算函数、Agent 调用全部在同一进程内运行，不存在跨语言通信开销。FastAPI 的类型注解体系与工具集的参数校验逻辑高度一致，工具层函数可以以极低的改造成本直接暴露为 API 端点。

后端的路由层划分为四组，与工具集的分类对应：`/meta` 负责元信息读写，`/data` 负责表数据读写，`/compute` 负责公式执行和算法 API 调用，`/validate` 负责验证与分析。Agent 的调度入口单独作为 `/agent` 路由组，接收自然语言请求或结构化任务描述，返回流式输出以支持前端实时展示执行过程。

Agent 的执行全部在后端进程内完成，前端不直接接触任何 LLM 调用逻辑。这一设计保证了 API Key 不暴露在前端，同时让 Agent 的工具调用链在后端形成完整的可追踪日志。

前后端通信全部使用 JSON over HTTP，Agent 流式输出使用 Server-Sent Events，其余接口均为标准 REST，不引入 WebSocket。

---

## 前端：React

前端使用 React 构建，以 Vite 作为开发和构建工具。选择 React 而非 Streamlit 的核心原因是交互复杂度：数值工具需要支持行内单元格编辑、手动标记保护状态、多表并排查阅、README 侧边栏随时展开收起、Agent 执行日志实时滚动——这些交互在 Streamlit 中实现代价极高，在 React 中属于基础能力。

UI 布局分为三个主要面板。左侧是项目导航面板，展示所有表的列表、每张表的当前验证状态（通过 / 警告 / 未验证），以及快速跳转入口。中间是主工作区，以类电子表格形式展示当前选中表的数据，每列顶部显示来源标记的统计摘要，支持行内编辑并在编辑时自动标记来源为用户手动输入。右侧是上下文面板，展示当前表的 README 文档，并在 Agent 执行任务时切换为实时执行日志视图。

Agent 交互入口位于页面底部，用户输入自然语言请求，前端通过 SSE 接收后端的流式响应，实时渲染 Agent 的每一步操作日志。执行完成后日志区折叠，变更摘要以卡片形式固定展示。

前端在构建后生成静态产物，由 Nginx 直接 serve，不依赖 Node 进程常驻。

## 数据层：SQLite

SQLite 以单文件形式存储所有项目数据，每个项目对应一个独立的 `.db` 文件，存放在服务器上的指定数据目录中。SQLite 在这个场景下完全够用：写入操作是小批量串行的，没有并发写入压力；读取操作即使全表扫描也在毫秒级完成；文件体积在游戏数值场景下不会超过几十 MB。

数据库内部除存储业务数据表之外，还维护四张系统表：`_table_registry` 存储所有动态层表的元信息和 README；`_dependency_graph` 存储表间和列间的依赖关系；`_formula_registry` 存储所有已注册的公式字符串；`_snapshots` 存储每次重大变更的数据快照引用。这四张系统表是整个工具运转的骨架，Agent 的所有规划和推导都依赖它们，而不是依赖对业务数据表的直接扫描。

业务上的「固定层 / 动态层」、单元格来源标记、表级 README 的维护方式见 `05`。

## 服务器部署架构

服务器上的进程层级分为三层：**Nginx 作为反向代理**处理所有外部请求，**Uvicorn 运行 FastAPI 后端**，静态前端产物由 Nginx 直接 serve。

Nginx 的职责是：将 `/api/*` 路径的请求代理到 Uvicorn 进程，其余路径直接返回 React 构建产物的静态文件，同时负责 HTTPS 终止。由于是小团队工具，不需要多个 Uvicorn worker，单进程运行即可，反而更符合 SQLite 单写者的约束。

FastAPI 进程使用 **systemd** 管理，配置为系统服务，服务器重启后自动拉起，崩溃后自动重启。日志通过 journald 收集，`journalctl -u balance-tool -f` 即可实时查看后端日志。不引入 Docker 等额外层级，直接在宿主机的 Python 虚拟环境中运行，减少排查问题时的复杂度。

```
/etc/systemd/system/balance-tool.service
/var/www/balance-tool/          ← Nginx serve 的前端静态产物
/opt/balance-tool/              ← 后端代码和 Python 虚拟环境
/data/balance-tool/projects/    ← 所有项目的 .db 文件和配置
```

HTTPS 使用 Let's Encrypt 签发证书，通过 Certbot 自动续期。如果只在内网或 VPN 内访问，也可以使用自签名证书。

## 访问控制

由于工具部署在公网可访问的服务器上，需要基本的访问控制。认证方案选择最轻量的 **HTTP Bearer Token**，在 FastAPI 中间件层统一校验，Token 存储在服务器的环境变量中。前端在首次打开时展示 Token 输入框，通过后 Token 存储在浏览器 localStorage，后续请求自动携带。

不引入完整的用户系统和权限管理，原因是用户规模不需要——这是个人或固定小团队工具，Token 认证足以防止公网扫描和误访问。如果未来需要多用户隔离，再在此基础上扩展，而不是一开始就过度设计。

LLM 的 API Key 存储在服务器的 `.env` 文件中，由后端进程在启动时加载到环境变量，不出现在任何 API 响应或前端代码中。

> **与 `01` 的说明**：若产品仍要求「账号密码 + 邀请码」的多用户模型，可在本节的 Token 或会话层之上实现独立认证模块；本节描述的是 **当前文档撰写时的轻量部署形态**。

## 项目管理与文件结构

每个游戏项目在服务器文件系统上对应一个独立目录，目录内包含项目数据库文件 `project.db`、项目级配置文件 `config.json`（存储算法 API 注册表、验证规则集等不适合存入数据库的配置），以及一个 `exports` 子目录用于存放导出的 CSV 或 Excel 快照。

项目目录统一存放在 `/data/balance-tool/projects/` 下，由后端统一管理路径，前端通过项目 ID 引用，不暴露服务器文件路径。多项目切换在前端通过项目选择界面实现，后端在同一进程内按请求携带的项目 ID 加载对应数据库连接，不需要重启服务。

数据备份方面，由于项目数据是单文件 SQLite，直接对 `/data/balance-tool/projects/` 目录做定时 `rsync` 或 `tar` 备份即可，也可以纳入服务器整体的快照备份策略。如果需要更细粒度的变更历史，工具提供 `export_changelog` 命令，将 `_snapshots` 表的变更记录导出为 Markdown 文件，方便手动归档。
