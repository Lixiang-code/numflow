## 待办 1：gameplay_table "进行中" 孤儿状态恢复

### 问题描述

`set_gameplay_table_status` 工具将表设为「进行中」后，如果 Agent 崩溃/中断，
该表永远停在「进行中」，后续 Agent 的 `get_gameplay_table_list` 查询会跳过它（非 未开始/待修订），
导致该表被永久遗漏，整个流水线无法完成。

### 设计方案

**方案 A：超时自动重置（推荐）**

在 `_gameplay_table_registry` 表增加 `started_at TIMESTAMP` 字段，
`set_gameplay_table_status` 写入「进行中」时同时更新 `started_at = now()`。

`get_gameplay_table_list` 返回时，自动将满足以下条件的表重置为「未开始」：
- `status = '进行中'`
- `started_at < now() - interval '2 hours'`（超时阈值，可配置）

这样下一个 Agent 轮询时会自然看到该表并重新处理。

**涉及文件：**
- `backend/app/core/project_schema.py` — `_gameplay_table_registry` 表结构加 `started_at`
- `backend/app/core/project_migrations.py` — 迁移：存量表 ALTER COLUMN 加 `started_at`
- `backend/app/services/agent_tools.py`：
  - `set_gameplay_table_status`：进入「进行中」时写 `started_at`
  - `get_gameplay_table_list`：查询前先执行超时重置（UPDATE status='未开始' WHERE 超时）

**方案 B：前端手动重置按钮**

在 ProjectSetup 的玩法表列表旁加「重置为未开始」按钮（仅「进行中」状态可见），
对应后端新增 `PATCH /pipeline/gameplay-tables/{table_id}/reset` API。

简单但需要人工干预，不如方案 A 自动化。

**建议：两者都做，A 做自动恢复，B 做人工兜底。**

## 待办 2：结束审核阶段的限制

### 问题描述

当 execute 轮数很多（现在无限制）时，结束审核逻辑上是正常的。
但如果 AI 在结束审核中发现严重问题，当前只有一次补救机会，
审核后无论如何都会 emit done 退出。

### 是否需要处理

暂不处理，观察真实测试中是否出现问题。若需要，可考虑：
- 结束审核阶段发现问题 → AI 调写工具修正 → 修正完后输出总结（已支持，因为审核阶段仍在 execute 循环中）
- 目前唯一限制是：结束审核后第二次无工具调用就直接退出

**结论：现有实现已足够，AI 在结束审核时可以继续调工具修正，只有再次无工具调用才退出。**
