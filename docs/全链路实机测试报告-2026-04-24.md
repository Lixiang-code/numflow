## 1. 测试目标

1. **流程通畅性**：注册登录 → 项目列表/新建 → 带 `X-Project-Id` 的元数据/数据/流水线/计算/验证 → Nginx 聚合访问。
2. **Agent 效果**：千问连通性、显式缓存诊断、SSE 流式对话、Function Calling 调工具（`get_project_config` / `get_table_list`）。
3. **与文档一致性**：流水线顺序校验、模板项目写权限、工具层统一返回结构（通过 Agent 预览 JSON 观察）。
4. **客户端（React）**：构建与静态资源、SPA 深链、`Workbench` 首屏所需 API 序列、与后端契约一致性（对齐 [01-产品与界面需求](./游戏数值系统AI化自动开发-01-产品与界面需求.md) 与 `04` 三面板 + Agent 条描述）。

## 3. Agent 行为观察（效果评价）

- **工具选择**：在用户明确要求「只调用一次 get_project_config」时，首轮即调用该工具，第二轮输出结论，符合文档 `06` 中「先读配置再作答」的预期。

- **工具返回预览**：`tool_result.preview` 截断展示 JSON 片段，便于排查；完整结果在模型侧 message 中继续传递。

## 4. 发现的问题与建议

### 4.1 运行中代码与仓库源码可能不一致（务必在发版后重启 Uvicorn）

**现象（重启前）**

- SSE 首条日志为旧文案「维护模式」等，与仓库 `agent_runner.py` 不一致。
- `openapi.json` 中缺少 `GET/POST /api/meta/snapshots`、`GET /api/meta/default-rules`、`PUT .../validation-rules` 等，导致 **工作台首屏** 依赖的 `GET /meta/snapshots` **404**，整页加载失败。

**处置（本次测试）**

- 已对本机 `127.0.0.1:8000` Uvicorn **重启**；重启后 OpenAPI 含快照相关路由，`mode=init` 时日志为「初始化 Agent」，**客户端工作台 API 序列通过**。

**建议**

- 每次合并后端代码后执行 `systemctl restart <服务名>` 或重启 Uvicorn，并在发布 checklist 中固定「**核对 openapi 是否含 Workbench 依赖路径**」一项，避免再次出现「前端已发布、后端仍为旧进程」的半升级状态。

### 4.2 systemd 服务名与文档

文档 `04` 示例服务名为 `balance-tool`，本机该 unit 为 **inactive**，实际由 `uvicorn` 直接监听 `8000`。  

**建议**：统一文档中的服务名与 aaPanel/实际 unit，或补充「aaPanel 场景下进程管理方式」说明，避免排障时误判。

### 4.3 诊断接口的 token 用量

`diagnostics/run` 的连通性用例中，`usage.completion_tokens` 较高（模型仍生成较长回复），不影响功能，若需降本可在诊断提示词中进一步收紧 `max_tokens`（属优化项）。

### 4.4 前端 ESLint 未通过（技术债）

`npm run lint` 曾在 `NewProject.tsx` 报错（草稿 `useEffect` 内同步 `setState`）。**§6 已重构为 `useMemo` 初值 + `useCallback` 持久化**，`lint` 通过；仍建议在 CI 中固定执行 `npm run lint`。

### 4.5 登录页默认填充账号（安全提示）

`Login.tsx` 曾内置默认用户名与密码初值；公网部署存在误导与误用风险。**§6 起已改为空初值**；本地联调请自行输入测试账号或查阅运维侧账号说明。
