# 05 — 数据层与计算设计

> 实现 `01`～`03` 所需的 **领域与数据模型**：固定层/动态层、单元格来源、README、公式、算法 API、增量调度。  
> **呈现层与路由划分**见 `04`（FastAPI `/meta` `/data` `/compute` `/validate` 与前端三面板）。  
> Agent 角色、提示词与工具清单见 `06`。

## 实现落地与领域设计（数据与计算）

### 技术架构分层（数据 / 算法 / Agent）

整个工具在技术层面分为三个计算层。数据层使用 SQLite 存储所有可以表格化的内容，包括项目配置参数、静态定义数据、以及所有由 AI 或算法生成的结果值。算法层将所有无法用表格直接表达的计算逻辑封装为 Python 函数并暴露为内部 API，例如马尔科夫链状态转移、复杂概率期望、递归式的多级防御传递计算等；这些 API 接收明确参数并返回结构化结果，计算结果统一回写入数据层以便后续复用。Agent 层作为最上层的调度核心，工具集同时覆盖数据层的读写操作和算法层的 API 调用，由 Agent 自主判断某个计算任务应当查表、写表，还是调用算法 API。

用户可见的界面由 **React 前端** 实现（布局与交互见 `04`），通过 HTTP API 访问上述能力；不在浏览器内直接执行 Pandas 或 Agent 逻辑。

### 动态表结构与元数据驱动

由于不同项目的表数量和表结构完全由项目配置决定，工具不采用硬编码 schema，而是元数据驱动的方式。数据库内分为固定层和动态层：固定层存储每个项目都具备的配置信息，包括核心定义、游戏系统开关、属性系统定义等，结构相对稳定；动态层的所有计算表由 Agent 在项目初始化阶段根据固定层配置推导生成，Agent 负责实际执行建表操作，表数量与表结构随项目而变。

系统表 `_table_registry`、`_dependency_graph`、`_formula_registry`、`_snapshots` 的用途见 `04` 数据层小节；本节约定其 **业务语义** 与读写约束与 Agent 工具集一致。

每个单元格在写入时附带来源标记，区分"用户手动输入"、"AI 生成"和"算法派生"三种来源。Agent 在执行增量更新时默认不覆盖用户手动输入的单元格，算法派生数据与原始输入数据严格隔离，避免派生结果被误用为上游输入。

### 表级 README 机制

每张数据表对应一份 Markdown 文档（README），内容包括用途说明、列定义、依赖关系、以及当前维护状态。README 的存储与索引由 `_table_registry`（或等价元数据）承载；前端在上下文面板中打开查阅，同时也是 Agent 的上下文输入——Agent 在读取任何表之前先读取其 README，在写入或修改表之后自动更新对应 README，记录本次变更的内容与原因。README 是整个工具的活文档层，人和 AI 共同维护、共同消费。

### 命名引用与公式系统

工具内置一套命名引用语法，格式为 @表名[列名]，用于在公式字符串中跨表引用数据。例如：

@装备表[攻击力基数] * (1 + @属性表[攻击加成])

公式以字符串形式存储在数据库中，执行时由解析器将引用替换为对应的 Pandas Series 或标量值，在受控命名空间中通过 eval() 执行，不暴露任何系统级内置函数。执行前对所有公式进行依赖图的拓扑排序，检测并拦截循环依赖。Pandas 的广播机制自动处理标量与列向量的混合运算，无需额外适配。

算法层 API 调用统一纳入同一套公式语法，以 @API 前缀标识，例如：

@API[马尔科夫增幅期望](等级, 层数)

解析器识别到 @API 前缀时走 API 调用路径而非数据库查询路径，对用户和 AI 而言语法保持一致。整套公式系统属于一次性实现、长期稳定复用的基础设施。

### 算法 API 注册规范

算法层的每个 API 需要携带机器可读的元描述，内容包括接收的参数名称与含义、返回值的数据结构、以及适用场景的自然语言说明。这是 Agent 能够在无硬编码的情况下自主选择调用哪个 API 的前提。

判断一个计算是否应当写成算法 API 而非表格公式的基本原则是：如果该计算存在状态依赖、递归传递、概率分布迭代、或条件分支在逻辑上无法被扁平化为有限行列，则应当代码化。典型例子包括多级防御增幅的逐层传递与溢出处理、攻速在大于 1 和小于 1 区间的分段算法、以及任何需要马尔科夫链建模的增幅收益期望计算。

### Agent 工具集设计（概述）

Agent 的工具集划分为三类。读取工具包括：读取项目配置与全局定义、读取指定表的 README、读取指定表的全量数据或按条件切片、读取单元格值及其来源标记、读取表间依赖关系。写入工具包括：向指定单元格或批量向指定列写入值并附来源标记、写入或更新指定表的 README、修改全局配置参数。计算与验证工具包括：执行指定公式字符串并将结果回写数据库、调用算法层指定 API 并将结果回写数据库、触发指定表依据上游依赖关系的重算、运行内置验证规则并返回验证报告、计算指定等级区间的核心平衡指标。

### 增量维护与 Agent 调度

项目初始化完成后，所有后续更新均通过 Agent 调度执行，不存在固定的更新 pipeline。Agent 接收变更请求时需要获取三项输入：自然语言描述的变更内容、变更类型标签（配置变更、表数据变更、或文档变更），以及当前所有用户手动编辑的单元格保护列表。Agent 根据项目的表间依赖关系自行推导影响范围、确定更新顺序、判断是否需要重新调用算法 API，并在相关表的 README 中记录本次更新的变更说明。是否合理的判断发生在 Agent 层，工具本身不做强制拦截。
